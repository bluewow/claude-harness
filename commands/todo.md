## 경로 규약

`{PROJECT_ROOT}` = `${CLAUDE_PROJECT_DIR}` → 없으면 cwd 에서 위로 올라가며 `.claude/` 가 있는 디렉토리.
TODO 파일 = `{PROJECT_ROOT}/docs/TODO.md` · 완료 이력 = `{PROJECT_ROOT}/docs/CHANGELOG.md`.

## 흐름 한눈에

```
내가 던진 TODO (거칠어도 됨)
   └─ 코드·PRD 로 파생/누락/개선 검토 ─→ 질문으로 확정 ─→ 다듬은 항목을 TODO.md 에 머지
```

> **내가 던진 한 줄은 부족하거나 개선이 필요할 수 있다**는 전제로 받는다. 그대로 받아 적지 말고, 검토해서 되묻고 함께 완성한다. 가정으로 빈칸을 메우지 않는다.

## 이 커맨드가 관리하는 것 (필터)

> TODO.md 는 **순수 비즈니스 로직·제품 동작**만 담는다. 시드든 파생이든 매번 이 기준으로 거른다.

| | 들어감 ✅ | 빠짐 ❌ (TODO 에 적지 않음) |
|---|---|---|
| 성격 | 정책·산식·플로우·도메인 규칙·화면 동작 | 인프라·운영·코드품질 |
| 예 | 수습 비율 게이트, 전환 모달, 급여정책 UI, 교부 리스크 점검, 집계 제외 규칙 | 배포·env·외부 키/계약(PASS·알림톡·Firebase), 푸시 토큰, App/Universal Links, 리팩토링(DTO·queryKey·에러표준화·View 분해), 테스트 픽스처, 빌드 |
| 빠진 건 어디로 | — | [운영-가이드.md] · [앱-가이드.md] 가 SoT — 발견하면 TODO 가 아니라 거기로 라우팅 |

추가 원칙:
- **완료는 사용자가 `- [x]` 수동 체크**. 이 커맨드는 절대 자동으로 완료 처리하거나 항목을 지우지 않는다.
- **인지 우선** — 항목은 한 줄, 무엇을(동작) + 근거 `PRD §` + (필요 시) 핵심 파일. 표·장문·구현 이력 금지.

## Step 0: 시드 받기

- `$ARGUMENTS` = **내가 직접 쓴 할 일/의도**(문장이든 키워드든). 이게 다듬기의 **기준 시드**.
- 비어있으면 "무엇을 추가할까요?" 한 줄 받아 시드로 삼는다.
- `--quick` = 검토·질문 생략하고 **시드만 그대로 한 줄 추가**(다듬기 없음). 빠르게 적어둘 때.

## Step 1: 파생 · 누락 · 개선 검토 (코드 + PRD)

시드를 그대로 신뢰하지 말고 대조한다.

1. 루트 `CLAUDE.md` + `web/CLAUDE.md`(§"문서 갱신" 매핑) → 시드가 닿는 도메인·정책 파악.
2. 관련 **정책 SoT** 읽기 — `{PROJECT_ROOT}/docs/PRD-*.md` 의 §A(요구·수용기준)·§B(구현모델).
3. 코드 확인 — 시드를 실제로 하려면 **딸려오는 일(파생)** 이 뭔지: 데이터/마이그레이션·산식·상태전이·화면·권한·영향받는 다른 플로우. PRD §A 수용기준 대비 빠진 단계.
4. **시드 자체의 개선점** — 모호/과광범위/이미 일부 구현됨/우선순위 불명/엣지케이스 누락.
5. **비즈니스 로직 필터** 적용 — 파생 중 인프라성은 버리거나 운영/앱 가이드로(TODO 엔 안 넣음).

→ 산출: ① 시드 정제안  ② 파생 후보 N개  ③ 질문거리(빈칸·결정·범위)

## Step 2: 질문 라운드 (핵심 단계)

> 부족분을 메우는 단계. **코드/PRD 로 이미 답이 나오는 건 묻지 않는다.** 시드가 이미 명확하고 파생도 자명하면 질문을 최소화하거나 생략한다.

먼저 한 줄로 검토 결과를 요약(시드 + 파생 후보)한 뒤 `AskUserQuestion` 1회(최대 4문항 묶음). 묻는 축:

- 시드의 **범위·완료 기준(AC)** 확정
- 파생 후보 중 **포함 / 제외**
- **우선순위** (`P0` 출시차단 · `P1` 다음 · `P2` 백로그)
- 갈리는 **엣지케이스·정책 분기**

각 문항에 추천 옵션을 먼저 두고 `알아서 판단`(Other 자유 입력)도 연다. 답 후 결정적으로 더 필요하면 1회만 추가 질문.

## Step 3: 항목화

- 시드(정제) + 채택된 파생 → 한 줄: `- [ ] \`P?\` **무엇(동작)** — 보충 한 줄 · §B.x`
- 닿는 도메인 섹션(`## 도메인 · [PRD](파일)`)에 배치. 새 도메인이면 섹션 신설.

## Step 4: TODO.md 머지 (부분 갱신)

1. `TODO.md` 있으면 `Read`.
2. 머지 규칙:
   - 닿는 도메인 섹션만 손대고 무관한 섹션은 그대로 둔다.
   - **체크된 `- [x]` 는 무조건 보존**(문구 유지). 새 항목 추가·중복 병합.
   - 코드상 이미 구현된 듯한 미체크 항목은 자동 체크하지 말고 `⚠️ 구현된 듯 — 확인` 만 덧붙인다.
3. 헤더 날짜 `# TODO — 노무히어로 (YYYY-MM-DD)` 갱신.
4. 비면 헤더 블록부터 생성. 전체 재작성(Write) 금지 — 변경 섹션만 `Edit`(신규 생성일 때만 Write).

## Step 5: 보고

채팅에 짧게:

```
✅ TODO 다듬음 — [시드 요약]
- 시드 1 + 파생 N 추가 (질문 반영)
- 뺀 것(인프라/범위 밖): [있으면 한 줄, 없으면 "없음"]
- ⚠️ 이미 구현된 듯: [항목 또는 "없음"]
```
