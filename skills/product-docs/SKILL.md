# 제품 문서 관리 (product-docs)

**목적: 프로젝트의 정책·스펙을 표준 포맷의 md 로 유지해, Claude 가 그 문서를 읽고 정확하게 개발하도록 한다. 이 md 는 사람이 손으로 쓰지 않고 ★AI 가 작성·갱신★하는 정책 SoT(단일 진실 소스)이며, 개발자도 읽기 쉬운 형태다.**

기획자·디자이너의 러프 자료는 **입력원**일 뿐, 그것을 이 md 로 구조화(`intake`)한 뒤 개발하고, 구현 결정을 다시 문서로 되먹인다(`sync`).

```
[기획자 러프 자료] ──intake──▶ [정책 md: 원페이저·PRD §A·IA] ──develop──▶ [구현] ──sync──▶ [정책 md 최신화]
                                       ▲                                                       │
                                       └──────────────── 다음 자료가 오면 다시 intake ──────────┘
```

## 2계층 구조 (이식성의 핵심)
- **스킬**(이 파일) = 포맷 규격 + 프로세스. 모든 프로젝트 공통, 도메인 무관.
- **프로젝트 매니페스트**(`<문서홈>/_manifest.md`) = 그 프로젝트의 문서 목록 + 변경→문서 매핑 + 문서 홈 경로. 프로젝트마다 다름.

> 스킬은 매핑을 하드코딩하지 않는다. **항상 매니페스트를 먼저 읽고 그 선언을 따른다.** 특정 프로젝트의 도메인 고유명사·기능명은 매니페스트와 문서에만 존재하고, 이 스킬에는 절대 박지 않는다.
> 매니페스트 파일이 없는 프로젝트는 **프로젝트 `CLAUDE.md` 의 문서 체계·매핑 표를 매니페스트로 간주**한다(별도 파일을 새로 만들지 않는다 — 매핑 사본은 항상 1곳).

## 문서 홈
- 기본값 `.output/docs/` (정본·직접 편집). 매니페스트 `doc_home` 으로 변경 가능(기존 `docs/` 를 쓰는 프로젝트 등).
- `.output/docs` 는 산출물이지만 **문서 정본이므로 git 추적 유지**: `.gitignore` 에 `.output/` 가 있으면 `!.output/docs/` 예외를 추가한다.

## 모드 디스패치
호출: `/product-docs <mode>` 또는 자연어. mode 미지정 시 매니페스트 유무·입력으로 추론.

| 입력 신호 | 모드 |
|---|---|
| "init / 초기화 / 골격" · 매니페스트 없음 | **init** — 문서 골격·베이스라인 생성 |
| "이 기획안/PRD 초안/기능요청 문서에 반영, intake" | **intake** — 러프 기획자료 → 정책 md(스펙화) |
| "리스트업 / 인벤토리 / 결정 워크시트" | **intake** — 항목 전수 리스트업 → `리스트업-{주제}.md` |
| "sync / 동기화 / 갱신 / 변경 반영 / drift 점검" | **sync** — 구현 → 문서 되먹임 + 정합성 점검 |
| "왜 X · 정책 · 가이드 질문 / 조회" | **route** — 인용 기반 답변 |

## init — 문서 골격·베이스라인 생성
새 프로젝트면 빈 골격, **이미 진행된 프로젝트면 현행 코드를 역공학해 코드와 일치하는 정책 베이스라인**을 만든다.
1. **프로젝트 스캔** — 루트 빌드파일·`CLAUDE.md`·디렉터리 구조를 읽어 스택·타입 파악(`setup:init` 로직 재사용 가능). 이미 `<문서홈>/_manifest.md` 가 있으면 덮어쓸지 먼저 확인.
2. **코드 역공학(기존 프로젝트)** — 코드에서 끌어와 사실부(§B 등)를 채운다:
   - 라우트 → `IA-현행화.md`
   - 스키마/모델 파일 → `PRD-DB스키마.md` §2·§3
   - API·핵심 로직·상태머신 → 도메인 `PRD-{도메인}.md` §B
   - 스키마의 엔티티·관계 + PRD 들의 횡단 규칙 → `온톨로지.md`(개념 사전·관계 그래프·공리·용어)
   - `.env*` → `운영-가이드.md` §1
3. **§A·의도 확인** — §A(목적·요구·정책의 "왜")는 코드에 없으므로, 러프 자료가 있으면 반영 + 부족분만 AskUserQuestion 으로 확인(가정 금지, 비어있을 때만).
4. **생성** — 문서 홈에 `templates/` 를 치환 생성: `원페이저`·`운영-가이드`(항상), `IA-현행화`·`PRD-DB스키마`(활성 시), `온톨로지`(도메인 2개 이상일 때 — 1개짜리는 PRD 로 충분), 도메인별 `PRD-{도메인}`, `_manifest.md`(문서목록 + 도메인 + 변경→문서 매핑).
5. **`.gitignore` 예외** + **보고**(생성 목록).

→ 이후 개발은 `intake → develop → sync` 루프.

## intake — 러프 기획자료 → 정책 md (개발 스펙 만들기)
기획자·디자이너가 준 불완전한 자료(원페이저 초안·기능 설명·PRD 초안·회의록)를 **실행 가능한 스펙**으로 구조화한다. 사람은 자료만 주고 **md 작성은 AI 가 한다.**
1. **분류** — 자료가 어디에 해당하나: 신규 기능/도메인 → `PRD-{도메인}.md`, 비전·범위·로드맵 → `원페이저.md`, 화면 → `IA-현행화.md`, **항목 전수 나열 + 채워갈 결정**(과금·채택·우선순위·스코프 등) → `리스트업-{주제}.md`(생성 시 매니페스트 활성 문서에 등록, 결정 확정 후 원페이저·PRD 로 승격).
2. **구조화** — 러프 텍스트에서 **§A(목적·사용자스토리·요구·정책)** 를 추출해 표준 포맷으로 작성. **§B 는 비워두고 "구현 시 확정"** 표시. 신규 도메인이면 매니페스트의 도메인 목록 + 변경→문서 매핑에 등록하고, `온톨로지.md` 가 있으면 새 개념·관계·공리도 함께 반영(기존 공리와 충돌하면 이 단계에서 드러내고 질문).
3. **갭 질문** — 자료엔 없지만 정확한 개발에 필요한 빈칸(엣지케이스·우선순위·데이터 필드·분기·권한)을 AskUserQuestion 으로 확인. **가정 금지** — 이 단계가 "정확한 개발"의 핵심.
4. **수용 기준(AC)** — §A.3 각 요구에 "완료 판정 기준"을 명시해 개발·검증이 정책에 맞물리게.
5. **보고** — 어느 문서에 무엇이 반영됐는지 + 남은 갭/질문.

→ 이후 개발은 이 §A 를 계약(spec)으로 진행하고, 구현 결정(상태머신·산식·API)은 `sync` 가 §B 에 기록한다.

## sync — 구현 → 문서 되먹임 + drift 점검 (스킬의 심장)
구현하며 생긴 변경·결정을 정책 md 에 되먹이고, **코드가 정책과 어긋났는지** 점검한다.
1. **매니페스트 로드** — `<문서홈>/_manifest.md` 의 매핑표·문서 목록·`last_sync`.
2. **변경 감지** — git 이면 `git diff --name-only`(가능하면 `last_sync` 이후) + `git status`; git 아니면 최근 수정 파일(Glob 은 mtime 정렬) + **현재 대화에서 바꾼 것**. 매핑표에 걸리는 **정책 영향 변경**만 다룬다(무관한 코드 변경은 무시).
3. **되먹임(부분 갱신)** — 변경을 매핑표로 대상 문서·섹션에 대조해 **해당 섹션만 Edit**(주로 §B·IA·DB스키마·운영가이드). 애매하면 한 줄로 확인.
4. **drift 점검** — 매핑된 영역에서 **코드와 정책 md 가 어긋난 곳**을 찾아 리포트:
   - ① 코드엔 있는데 정책 문서에 없음(미문서화)
   - ② 정책엔 있는데 코드가 다르게 구현됨(정책 위반)
   - ③ §A 요구인데 미구현
   각 건을 "어디서·무엇이·어떻게 다른가"로 보고하고, **문서를 고칠지 / 코드가 틀린 건지** 사용자에 확인(임의로 어느 쪽도 진실로 단정하지 않음).
5. **스키마 변경 시** — 마이그레이션 + `PRD-DB스키마.md` §3 행 + 해당 도메인 PRD **3곳 동시** 갱신. 개념·관계·횡단 규칙이 바뀌었으면 `온톨로지.md`(있으면)의 해당 섹션도.
6. **보고** — `변경 / drift → 손댄 문서·섹션 + 미해결 항목`을 요약. 가능하면 매니페스트 `last_sync` 갱신.

## route — 조회 (인용 기반 답변)
1. 매니페스트 매핑으로 질문 → 우선 문서 1~2개 라우팅.
2. Grep 으로 위치를 찾고 매치 **±20줄만** 부분 Read(전체 읽기 금지, 200줄 미만일 때만 예외).
3. `파일:줄번호` 인용을 포함해 답한다. 문서에 없으면 추측하지 말고 "문서에서 확인되지 않음"으로 명시.

## 공통 가독성 규격 — 모든 산출 md

기준은 "**심플이 기본, 디테일은 정책 문서에만**". init·intake·sync 가 만들거나 고치는 모든 md 에 적용한다.

1. **머리 blockquote** — 목적·사용법(누가 언제 여나)을 1~2줄만 문서 맨 위 `>` 로. 그 이상의 서문 금지.
2. **상태 표시는 체크박스만** — `[ ]`/`[x]` 외 이모지·아이콘·범례 표 **금지**. 속성·강조·상태는 **굵은 텍스트 라벨**로 쓴다(예: **공개**, **로그인**, **스텁**, **미구현**).
3. **표는 5컬럼 이하** — 행마다 반복되는 공통 정보(SoT 링크 등)는 컬럼이 아니라 그룹 헤더 blockquote 로 뺀다. 셀 안 설명은 한 줄, 길어지면 표 밖 프로즈로.
4. **결정된 것 / 결정할 것 분리** — 확정 사항(날짜 명기)과 열린 질문(논의 가이드)을 별도 섹션으로 나눈다.
5. **디테일 이원화** — 구현의 근거가 되는 정책 문서(PRD §A/§B·DB스키마·IA)만 상세하게 쓴다. 그 외(백로그·체크리스트·이력·가이드)는 체크박스 리스트 + 최소 프로즈로 유지하고, 요약표·범례·부속 섹션을 늘리지 않는다.

## 절대 금지
- ❌ 문서 전체 재생성(Write). 변경분만 Edit 한다.
- ❌ 한 변경 때문에 무관한 섹션·다른 문서까지 손대기.
- ❌ 매핑을 이 스킬에 하드코딩. 항상 매니페스트가 SoT.
- ❌ 도메인 고유명사를 템플릿/스킬에 박기.
- ❌ 러프 자료의 빈칸을 임의 가정으로 메우기 — intake 단계에서 질문으로 확정.

| 논의 | `리스트업-{주제}.md` | 항목 전수 리스트업 + `[ ]` 결정 워크시트 — 확정 시 원페이저·PRD 로 승격 |

**§A/§B 규율**: §A 는 코드 용어(enum·FSM·엔드포인트) **금지**(공용어), §B 만 기술어. §A.3 요구에는 **수용 기준(AC)** 을 명시한다. §B 는 intake 시 비우고 구현·`sync` 시 채운다.

## 유지보수
- 새 PRD 추가/도메인 변경 시: 매니페스트의 **문서 목록 + 변경→문서 매핑**을 같은 작업에서 갱신.
- 매니페스트 매핑이 곧 머지 조건: 정책·플로우·스키마가 바뀐 변경은 대응 문서에 변경이 보여야 한다.
- 자동 강제가 필요하면(커밋/변경마다 문서 동기화 알림) Claude Code **hook** 으로 `sync` 안내를 거는 것을 검토(스킬 자체는 호출 기반).
