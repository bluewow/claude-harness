# 제품 문서 관리 (product-docs)

**목적: 어느 프로젝트에서나 동일한 표준 포맷의 제품 문서(원페이저·PRD·IA·DB스키마·운영가이드)를 생성·갱신·조회한다. 변경점을 감지해 ★변경된 부분만★ 부분 갱신하고, 문서 전체를 재생성하지 않는다.**

## 2계층 구조 (이식성의 핵심)
- **전역 스킬**(이 파일) = 포맷 규격 + 프로세스. 모든 프로젝트 공통, 도메인 무관.
- **프로젝트 매니페스트**(`<문서홈>/_manifest.md`) = 그 프로젝트의 문서 목록 + 변경→문서 매핑 + 문서 홈 경로. 프로젝트마다 다름.

> 스킬은 매핑을 하드코딩하지 않는다. **항상 매니페스트를 먼저 읽고 그 선언을 따른다.** 특정 프로젝트의 도메인 고유명사·기능명은 매니페스트와 문서에만 존재하고, 이 스킬에는 절대 박지 않는다.

## 문서 홈
- 기본값 `.output/docs/` (정본·직접 편집). 매니페스트 `doc_home` 으로 변경 가능(기존 `docs/` 를 쓰는 프로젝트 등).
- `.output/docs` 는 산출물이지만 **문서 정본이므로 git 추적 유지**: `.gitignore` 에 `.output/` 가 있으면 `!.output/docs/` 예외를 추가한다.

## 모드 디스패치
호출: `/product-docs <mode>` 또는 자연어. mode 미지정 시 매니페스트 유무로 추론.

| 입력 신호 | 모드 |
|---|---|
| "init / 초기화 / 골격 / 문서 만들어" · 매니페스트 없음 | **init** |
| "sync / 동기화 / 갱신 / 최신화 / 변경 반영 / PRD 업데이트" | **sync** |
| "왜 X · 정책 · 가이드 질문 / 조회" | **route** |

## init — 문서 골격 생성
1. **프로젝트 스캔** — 루트 빌드파일·`CLAUDE.md` 를 읽어 스택·타입 파악(`setup:init` 로직 재사용 가능). 이미 `<문서홈>/_manifest.md` 가 있으면 덮어쓸지 먼저 확인.
2. **최소 Q&A** (AskUserQuestion) — 한 줄 정의·페르소나·핵심 기능/도메인 목록·활성 문서(IA / DB스키마 on·off). **CLAUDE.md 등에서 답이 찾히면 묻지 않는다**(가정은 금지, 비어있을 때만 질문).
3. **생성** — 문서 홈 폴더를 만들고 `templates/` 의 각 파일을 복사해 `{{PLACEHOLDER}}` 를 실제 값으로 치환:
   - `원페이저.md`, `운영-가이드.md` — 항상
   - `IA-현행화.md`, `PRD-DB스키마.md` — 활성 시
   - 도메인별 `PRD-{도메인}.md` (도메인 수만큼)
   - `_manifest.md` — 활성 문서 목록 + 도메인 PRD + 변경→문서 매핑
4. **`.gitignore` 예외** — 위 문서 홈 규칙대로 처리.
5. **보고** — 생성된 파일 목록.

템플릿 위치: 이 스킬 폴더의 `templates/`.

## sync — 변경 감지 → 부분 갱신 (스킬의 심장)
1. **매니페스트 로드** — `<문서홈>/_manifest.md` 의 매핑표·문서 목록·`last_sync`.
2. **변경 감지**
   - **git 저장소**: `git diff --name-only`(가능하면 `last_sync` 이후) + `git status`.
   - **git 아님/미초기화**: 최근 수정된 소스 파일(Glob 은 mtime 정렬) + **현재 대화에서 방금 바꾼 것**을 변경 영역으로 삼는다.
3. **매핑** — 변경된 파일/영역을 매니페스트 매핑표에 대조해 갱신 대상 문서·섹션을 도출. 애매하면 사용자에게 한 줄로 확인.
4. **부분 갱신** — 대상 문서의 **해당 섹션만 Edit**. (아래 "절대 금지" 준수)
5. **스키마 변경 시** — 마이그레이션 + `PRD-DB스키마.md` §3 행 + 해당 도메인 PRD **3곳 동시** 갱신.
6. **화면 변경 감지** — 화면정의서(HTML 스크린샷) 류는 직접 안 건드리고 "`screen-def` 스킬 실행 필요"라는 힌트만 띄운다.
7. **보고** — `변경 영역 → 손댄 문서·섹션`을 요약. 가능하면 매니페스트 `last_sync` 갱신.

## route — 조회 (인용 기반 답변)
1. 매니페스트 매핑으로 질문 → 우선 문서 1~2개 라우팅.
2. Grep 으로 위치를 찾고 매치 **±20줄만** 부분 Read(전체 읽기 금지, 200줄 미만일 때만 예외).
3. `파일:줄번호` 인용을 포함해 답한다. 문서에 없으면 추측하지 말고 "문서에서 확인되지 않음"으로 명시.

## 절대 금지 (부분 갱신 규율)
- ❌ 문서 전체 재생성(Write). 변경분만 Edit 한다.
- ❌ 한 변경 때문에 무관한 섹션·다른 문서까지 손대기.
- ❌ 매핑을 이 스킬에 하드코딩. 항상 매니페스트가 SoT.
- ❌ 도메인 고유명사를 템플릿/스킬에 박기.

## 표준 문서 골격 (TODO·앱 가이드 제외)
| 분류 | 파일 | 역할 |
|---|---|---|
| 최상위 | `원페이저.md` | 비전·페르소나·로드맵·KPI |
| 화면 | `IA-현행화.md` | 화면·라우트 카탈로그 |
| 도메인 | `PRD-{도메인}.md` | §A 제품요구(무엇/왜) + §B 구현모델(어떻게) |
| 데이터 | `PRD-DB스키마.md` | 횡단 데이터 모델 SoT (도메인 PRD가 참조) |
| 운영 | `운영-가이드.md` | env·실행·배포 |

## 유지보수
- 새 PRD 추가/도메인 변경 시: 매니페스트의 **문서 목록 + 변경→문서 매핑**을 같은 작업에서 갱신.
- 매니페스트 매핑이 곧 머지 조건: 정책·플로우·스키마가 바뀐 변경은 대응 문서에 변경이 보여야 한다.
- 자동 강제가 필요하면(커밋/변경마다 문서 동기화 알림) Claude Code **hook** 으로 `sync` 안내를 거는 것을 검토(스킬 자체는 호출 기반).
