# Design Inspiration

이 스킬은 모든 프론트엔드 디자인 작업이 사용자가 선호하는 디자인 스타일과 시각적 일관성을 유지하도록 보장합니다. UI 코드를 생성하거나 수정하기 전에 디자인 시스템 문서(`references/DESIGN.md`)와 업로드된 인스피레이션 이미지를 참조합니다.

## 활성화 조건

- 새 페이지, 컴포넌트, 화면 생성 시
- 기존 CSS, 스타일, 시각적 레이아웃 수정 시
- UI 요소(카드, 버튼, 헤더, 폼 등) 제작 시
- 프론트엔드 인터페이스 리팩토링 또는 리디자인 시
- 시각적 UI 결과물이 포함된 모든 작업

---

## 워크플로우

### Step 0: 디자인 시스템 문서 확인 및 로드

디자인 작업 전, `references/DESIGN.md`를 읽습니다.

**분기 처리:**

- **DESIGN.md가 비어있거나 "아직 디자인 시스템이 생성되지 않았습니다"인 경우** → **[디자인 시스템 생성]** 워크플로우로 이동
- **DESIGN.md가 존재하고 내용이 있는 경우** → Step 1로 진행

### Step 1: 레퍼런스 이미지 로드

`images/` 디렉토리에 있는 모든 이미지 파일(PNG, JPG, WebP, SVG)을 Read 도구로 확인합니다.

`images/` 폴더가 비어있으면 사용자에게 알립니다:
> "Design Inspiration 폴더에 레퍼런스 이미지가 없습니다. 원하는 디자인 스타일의 이미지를 추가하면 더 정확한 디자인이 가능합니다."

이미지 없이도 DESIGN.md 기반으로 작업을 진행합니다.

### Step 2: 디자인 패턴 분석

DESIGN.md 규칙과 레퍼런스 이미지 인사이트를 결합하여 통합 디자인 방향을 수립합니다.

**DESIGN.md 기반 (항상 적용):** 문서에 정의된 색상, 타이포그래피, 간격, 컴포넌트 규칙을 엄격히 따릅니다.

**레퍼런스 이미지 기반 (보충):** 색상 액센트, 간격 리듬, 컴포넌트 형태, 시각 효과, 전체 톤을 참고합니다.

### Step 3: 결과물에 적용

1. **시각적 톤을 맞춤** — 레퍼런스를 픽셀 복사하지 않고 동일한 디자인 언어 사용
2. **기존 프로젝트 토큰 재사용** — `globals.css`의 토큰이 맞는 경우 활용
3. **새 토큰은 필요할 때만 제안**
4. **일관성 유지** — 같은 세션 내 모든 컴포넌트 간 시각적 통일

### Step 4: 디자인 결정 설명

코드 생성 후, 어떤 레퍼런스(DESIGN.md 규칙 또는 이미지)가 주요 결정에 영향을 주었는지 간략히 설명합니다.

## 디자인 시스템 생성 (DESIGN.md가 비어있을 때)

DESIGN.md가 비어있거나 초기 상태인 경우, 다음 순서로 디자인 시스템을 자동 생성합니다.

### 생성 Step 1: 소스 수집

프로젝트 상태에 따라 수집 가능한 소스가 다릅니다:

**A. 프로젝트 코드가 존재하는 경우:**
1. **CSS 변수/토큰 파일**: `globals.css`, `variables.css`, `theme.css` 등을 검색
2. **기존 컴포넌트**: 프로젝트 내 UI 컴포넌트의 스타일 패턴 분석
3. **패키지/프레임워크**: `package.json`, `tailwind.config`, MUI theme 등에서 디자인 시스템 단서 수집
4. **레퍼런스 이미지**: `images/` 디렉토리의 이미지를 `analysis-guide.md` 기준으로 분석

**B. 프로젝트가 아직 없는 경우 (빈 디렉토리 또는 초기 세팅만 존재):**
1. **레퍼런스 이미지**: `images/` 디렉토리의 이미지가 유일한 자동 수집 소스
2. **사용자 입력**: 디자인 방향, 업종, 톤, 참고 사이트 등을 사용자에게 직접 질문

이미지도 없고 프로젝트 코드도 없는 경우:
> "디자인 시스템을 생성하려면 최소한 다음 중 하나가 필요합니다:
> 1. `images/` 폴더에 레퍼런스 이미지 추가
> 2. 원하는 디자인 방향을 설명 (예: '미니멀 SaaS 대시보드', '럭셔리 이커머스')
>
> 어떤 방식으로 진행할까요?"

### 생성 Step 2: 사용자 확인

수집한 정보를 요약하여 사용자에게 디자인 방향을 확인합니다:
> "다음 정보를 기반으로 디자인 시스템을 생성합니다: [요약]. 추가 요구사항이 있으면 알려주세요."

### 생성 Step 3: DESIGN.md 작성

아래 **[DESIGN.md 표준 포맷]** 에 맞춰 `references/DESIGN.md`에 작성합니다.

## 디자인 시스템 갱신 (코드 변경 시)

다음 상황에서 DESIGN.md 갱신이 필요합니다:

- 사용자가 "디자인 시스템 업데이트해줘", "디자인 동기화해줘" 등 명시적 요청
- 새로운 레퍼런스 이미지가 추가되어 방향 변경이 필요할 때
- 사용자가 기존 디자인 방향과 다른 스타일을 요청할 때

### 갱신 프로세스

1. **현재 DESIGN.md 로드** — 기존 내용 확인
2. **변경 사항 식별** — 새로운 이미지, 사용자 요청, 코드 변경 내용 파악
3. **차이점 분석** — 기존 규칙과 충돌하는 부분, 추가할 부분 식별
4. **사용자 확인** — "DESIGN.md에 다음 변경을 적용합니다: [변경 목록]. 진행할까요?"
5. **DESIGN.md 업데이트** — 표준 포맷 유지하며 해당 섹션만 수정

**주의:** 갱신 시 전체를 다시 쓰지 않고, 변경이 필요한 섹션만 수정합니다.

## 우선순위 (충돌 해결)

소스 간 충돌 시:

1. **`references/DESIGN.md`** — 공식 디자인 시스템 (최고 우선순위)
2. **`images/` 레퍼런스 이미지** — 시각적 방향 보충 자료
3. **`globals.css`** — 기존 프로젝트 토큰
4. **CLAUDE.md 스타일 가이드** — 일반 프로젝트 규칙

**신규 컴포넌트**: DESIGN.md를 엄격히 따르고, 이미지로 보충합니다.
**기존 컴포넌트 수정**: 사용자가 명시적으로 리디자인을 요청하지 않는 한 현재 토큰을 유지합니다.
